BoltPage · UI Redesign Proposal

A quieter, native-feeling BoltPage

A redesign aligned to Apple's Human Interface Guidelines for macOS: smaller toolbar, system materials over decorative gradients, native typography, and a calmer document surface so the content does the talking.

BoltPage today reads as a web app dressed in glass — heavy blurs, large rounded panels, prominent floating shadows, custom pill buttons, and gradient backgrounds. macOS apps in the Sonoma / Sequoia era go the other direction: chrome recedes, materials are semantic, and toolbar items are compact icon-plus-label controls grouped by function. This proposal applies those principles to BoltPage's viewer, editor, and ancillary surfaces (find, view popover, welcome) without changing what the app does.

1 · Principles being applied

Chrome recedes

"Remove background colors from custom toolbars and tab bars. Rely on layout and grouping to express hierarchy" (HIG, 2025 update). No more shadowed, rounded-rectangle floating toolbars.

Semantic materials

Use materials by purpose: titlebar/toolbar material on the toolbar, sidebar material on the TOC, plain content on the document. Don't apply blur for decoration.

System typography

SF Pro for UI (system stack), New York / Iowan for the document. Avenir Next is removed: it is not the macOS system face and it adds to the "web-app" feel.

Compact, grouped toolbar

26pt-tall icon+label buttons with 6pt corners, separated into logical groups (file · doc actions · search) instead of one flat row of large pills.

Document is the hero

Drop the gradient background, drop the inner border + shadow on the document panel. The document sits on the window background; chrome only appears at edges (toolbar, sidebar).

Quieter onboarding

Replace the large display headline + eyebrow welcome with a centered icon, a one-line tagline, two CTAs, and a recent-files list — closer to Pages or Sketch's first-launch state.

2 · What's wrong today (audit)

Issues observed in current UI

  • Floating glass toolbar with 20px corner radius, heavy shadow, and 18px backdrop blur — reads as web/iOS, not Mac.
  • 42pt-tall pill buttons with both icon and label and 14px corners are roughly twice the visual weight of native NSToolbar items.
  • Brand icon embedded in the toolbar duplicates the Dock/title-bar identity.
  • Three nested rounded panels (toolbar 20px, sidebar 24px, content 28px) compound shadows and steal vertical space.
  • Decorative radial gradients on the body background have no semantic role and shift the perceived theme away from system appearance.
  • Welcome screen uses 3.25rem display type and a small-caps eyebrow — marketing landing-page tone, not first-launch tone.
  • "View" popover bundles theme + density + sidebar toggle in one custom dropdown; the sidebar caption ("Keep document context visible…") is over-wordy.
  • Find overlay floats at fixed pixel coordinates from the top — collides with toolbar at narrow widths and feels detached from the document it searches.
  • Editor toolbar duplicates the brand block and uses the same chunky pills; Lines / Wrap appear as plain text without icons or affordance.

What the redesign keeps

  • Three-theme support (light, dark, "drac") — translated into native palettes.
  • TOC sidebar with active-heading tracking, file metadata, collapsibility.
  • Same toolbar verbs: Open, New, Edit, Export, Find, Refresh, View options.
  • Find + Find/Replace in editor; Ctrl-F invokes a docked find bar instead of a floating overlay.
  • Type-size control, theme picker, sidebar toggle — all moved into a clean popover modeled on macOS view-options menus.
  • Markdown content typography and code-block styling unchanged in spirit; only token values are tuned.

Three densities so the user can pick whichever matches their screen size and habits. Default to icon + label on first launch (lowest ramp-up); allow switch via View popover.

7.2 View popover

The current "View" dropdown mixes appearance, type size, and a wordy sidebar toggle. The proposed popover splits these into clear sections, uses a segmented control for the appearance trio (matches macOS Settings > Appearance), a stepper for type size (mirrors System Settings > Display > Text Size), and a single switch for sidebar.

Captions are dropped — the labels are self-explanatory and the popover is invoked from a menu icon, not a button labeled "View".

7.3 Find bar (docked)

The current find overlay floats in the upper-right corner with a 16px-radius card and a 1300-z-index. It covers the document and detaches from the search context. The proposed find bar docks just below the toolbar — same place Safari, Pages, and TextEdit put it — and slides in/out with the document scroll-locked underneath.

8 · Recommendation summary

  1. Adopt Option B for the welcome state and the new toolbar geometry (26pt items, grouped, with separators) for both viewer and editor.
  2. Drop the brand icon from both toolbars. The window title carries identity; no need to repeat it.
  3. Replace floating panels with native materials: window background for content, sidebar material for the TOC, toolbar material under the title bar. No more 24/28px nested rounded cards.
  4. Move find from a floating overlay to a docked find bar under the toolbar; same in viewer and editor.
  5. Switch the type stack to -apple-system family for UI and "New York", "Iowan Old Style", Georgia for documents (current setup already uses Iowan as last fallback — promote it).
  6. Keep all three themes; retune their values to the proposed token palette so contrast and accent shift match macOS conventions.
  7. Add an editor inspector rail (word/char/cursor/encoding) — optional, off by default, exposes information that has no surface today.

Acceptance test: when the redesigned viewer launches against a 60-line Markdown file, the screenshot should be visually indistinguishable from a "vanilla" macOS document app (TextEdit / Pages / Preview lineage) at thumbnail size — toolbar, sidebar, and title-bar geometry should read as native chrome rather than as decorated panels.
